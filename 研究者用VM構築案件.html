1. まず押さえる前提（Azure VM 設計）

GPU の要否を先に確定

  • BindCraft は CUDA 対応 NVIDIA GPU が必須（CUDA バージョン指定も重要） GitHub
  • RFAntibody は Docker で GPU 利用（-gpus all）する想定 GitHub
  • BoltzGen も（内容次第で）GPUが実用上ほぼ前提になるケースが多いです（同系統の Boltz 系実装は pip install ...[cuda] を案内） GitHub

結論：この3点セットなら、基本は **Azure の GPU VM（N-series 等）**を前提にした方が手戻りが減ります。

OS は LTS（22.04 or 24.04）に寄せる

  • CUDA / NVIDIA driver / Docker 周りの情報量と安定性が高いです。
  • Azure 上での NVIDIA ドライバ導入は Microsoft の手順に寄せるのが安全です（ubuntu-drivers → ubuntu-drivers install → reboot など） Microsoft Learn

ディスク（永続領域）を最初から分ける

研究系は「コード」より「モデル重み・中間生成物・出力」が巨大化しがちです。推奨は：

  • OS ディスクとは別に データ用マネージドディスクを追加し、/data 等にマウント
  • /data/models /data/projects /data/outputs のように整理
  • スナップショット/バックアップ対象をデータディスク中心にする（OS は IaC / 手順書で再構築可能に）

2. Conda と Docker の“事故りやすいポイント”

Conda（Anaconda/Miniconda/Mamba）側の注意

  • 環境は“ツールごとに分離”（1つの conda 環境に全部入れない）
  • CUDA 系パッケージ（pytorch/cudatoolkit 等）は特に衝突しやすい
  • conda より mamba/micromamba の方が解決が速く、依存衝突の切り分けが楽なことが多い
  • 可能なら environment.yml を固定し、バージョンをピン留め（再現性の核心）

※ BoltzGen は「新規 conda 環境を作る」手順が明示されています（Python 3.12 の例） GitHub

Docker 側の注意

  • GPU をコンテナへ渡すために、ホスト側の NVIDIA driver と runtime 設定が必須
  • RFAntibody は docker run ... --gpus all ... を前提としています GitHub
  • ここで最も多い事故は「nvidia-smi はホストで動くが、コンテナ内で GPU が見えない」

    → 原因は NVIDIA Container Toolkit / runtime 設定漏れ、または driver/CUDA の不整合

「Conda + Docker 混在」の原則

  • ホスト側は“なるべく薄く”（OS更新 + Driver + Docker + 管理用ツール程度）
  • アプリ依存は、可能ならコンテナに閉じ込める
  • conda を使う場合も「ホストに直入れ」ではなく、ユーザ領域に分離して運用（root 汚染を避ける）

3. 今回（BoltzGen / BindCraft / RFAntibody）のベストプラクティス構成

おすすめは **「コンテナ優先＋例外のみ conda」**です。

推奨アーキテクチャ（実務向け）

  1. RFAntibody：Docker（原則固定）
  • 公式リポジトリでも Docker 実行例が提示されています GitHub
  • 実行時は v /data/projects:/home のように データ永続化は volume マウントで対応
  1. BindCraft：基本は GPU 前提。可能ならコンテナ運用（もしくは conda を完全分離）
  • BindCraft は CUDA バージョン指定の注意が明記されています GitHub
  • 「CUDA 11.8 / 12.x 等、VM の driver と整合するもの」を揃えるのが重要
  • もしプロジェクト側が Docker を出していない/弱い場合は、BindCraft 専用 conda env を作って隔離
  1. BoltzGen：プロジェクトの推奨に沿って conda env を分離
  • 公式の手順も “fresh environment” 前提 GitHub

運用面の“型”

  • /opt/apps：コード（Git clone）
  • /data：重み・入出力・作業ディレクトリ（永続）
  • conda env：~/miniconda3/envs/<tool名> などに分ける
  • docker images：タグ（または digest）を固定して、更新は計画的に

4. Azure VM 上での具体的な注意チェックリスト

GPU/Driver

  • ホストで nvidia-smi が正常
  • Microsoft の N-series Linux 手順に寄せて driver を入れる（ubuntu-drivers install など） Microsoft Learn
  • BindCraft 側で要求される CUDA バージョンを意識（間違うとインストールは通っても実行で落ちる） GitHub

Docker（GPU）

  • docker run --gpus all nvidia/cuda:... nvidia-smi が通る状態をゴールにする
  • RFAntibody のように -gpus all 前提のコンテナが動くことを確認 GitHub

セキュリティ/運用

  • SSH は鍵認証、必要なら Bastion / NSG で接続元制限
  • Docker の利用者を最小化（docker グループ＝実質 root 相当になり得る点に注意）
  • 更新方針を決める（OS/driver/CUDA/コンテナ/conda のどれをいつ更新するか）

再現性

  • conda は environment.yml（バージョン固定）を保管
  • コンテナは タグ固定ではなく digest 固定が理想（少なくとも使用バージョンを台帳化）
  • “動作確認済み構成”を README に残す（OS版本、driver、CUDA、GPU 型番、各ツールの commit/タグ）

5. 失敗しやすいポイント（よくある事故）

  • GPU VM ではなく CPU VM を作ってしまう（BindCraft で詰む） GitHub
  • NVIDIA driver とコンテナ側 CUDA の整合が取れていない
  • 1つの conda 環境に全部入れて依存関係が壊れる
  • 出力・重みを OS ディスクに置いて容量枯渇、または VM 作り直しで消失
  • “動いていたのにアップデートで壊れる”（タグ未固定、conda update 乱用）

必要であれば、次の情報が分かると「Azure の VM サイズ（GPU種別）」「CUDA/driver の組み合わせ」「Docker/Conda の切り分け」を含めて、手順をかなり具体化して提示できます。

  • 予定している Azure VM サイズ（例：NCas_T4_v3 / ND 系など）
  • Ubuntu のバージョン（22.04/24.04）
  • 使う想定の GPU（T4/A10/A100 等）と VRAM 要件の目安（対象タンパク質のサイズ感）
  • お客さんが求める提供形態（SSH で CLI 実行、Jupyter 提供、VS Code Remote 等）

condaは何をするものですか？

conda は一言で言うと、**「Python を含む実行環境そのものを、プロジェクトごとに安全に分離・管理するための環境管理ツール」**です。

単なる pip の代替ではなく、Python 本体・ライブラリ・ネイティブバイナリ（CUDA, MKL 等）まで含めて“環境一式”を管理できるのが最大の特徴です。

1. なぜ conda が必要になるのか（研究・AI系の現場視点）

今回のような：

  • BoltzGen
  • BindCraft
  • RFAntibody（これは Docker）

この手の 研究系・AI系ソフトは、ほぼ確実に以下の問題を起こします：

  • 「A は Python 3.10 必須、B は 3.12 必須」
  • 「A は torch 2.0 + CUDA 11.8、B は torch 2.4 + CUDA 12.x」
  • 「numpy のバージョン違いでどちらかが壊れる」

もし conda を使わずに：

pip install boltzgen
pip install bindcraft

のように 1つの OS 環境に全部突っ込むと、100% どれかが壊れます。

2. conda の本質的な役割

conda は次の3つを同時に管理します：

  1. Python のバージョン自体
  1. Python ライブラリ（numpy, torch, etc）
  1. ネイティブライブラリ（CUDA, cuDNN, MKL, libstdc++ など）

つまり：

「このソフト専用の仮想マシンっぽい環境」をユーザディレクトリ内に何個も作れる

というイメージです。

3. 何ができるか（具体例）

環境をツールごとに分ける

conda create -n boltzgen python=3.12
conda create -n bindcraft python=3.10

使うときに切り替える

conda activate boltzgen
python boltzgen.py


conda activate bindcraft
python bindcraft.py

それぞれ完全に独立した世界

  • boltzgen 環境の中の torch と
  • bindcraft 環境の中の torch は

同じマシンに存在しても、完全に別物として共存できます。

4. pip + venv との違い（重要）

項目venv + pipconda
Python のバージョン管理×○
CUDA / cuDNN×○
C/C++ 依存ライブラリ×（地獄）○
AI/科学計算系△◎

つまり：

AI / 機械学習 / 科学計算用途では、conda は事実上の標準インフラ

という位置付けです。

5. Docker との関係

  • Docker = OS から丸ごと環境を固める
  • conda = ユーザ領域で Python 実行環境を何個も持つ

役割が違います。

実務では：

  • RFAntibody → Docker
  • BoltzGen / BindCraft → conda

のように 併用するのが普通です。

全体像（まず結論）

この3つは 役割がはっきり分かれた「タンパク質設計パイプライン」 です。

設計 → 評価 → 抗体特化設計

という流れを分担しています。

典型的な流れ：

  1. BoltzGen：新しいタンパク質構造や配列の「候補」を大量生成する
  1. BindCraft：その中から「よく結合しそうなもの」を評価・最適化する
  1. RFAntibody：抗体に特化した設計・最適化を行う

① BoltzGen

■ 何ができるものか？

AI を使って「新しいタンパク質構造・配列」を生成するジェネレーター
  • 既存に存在しないタンパク質構造を「それっぽく」作る
  • 拡散モデル（Diffusion）系の生成AI
  • 「この形状のタンパク質が欲しい」という条件から構造を生成できる

要するに：

タンパク質の“設計案を大量に作る装置”

■ 想定される使い方

  • 研究者が：
    • 「このターゲットに結合しそうな骨格を作りたい」
    • 「このモチーフを含む構造を作りたい」
  • → BoltzGen に条件を与えて
  • → 数百～数千個の構造候補を生成
  • → PDB ファイルなどで出力

■ 入出力データ

入力：

  • 条件（ターゲット構造、モチーフ、拘束条件など）
  • 設定ファイル

出力：

  • タンパク質構造（PDB）
  • 場合によっては配列情報

■ 他アプリとの連携

  • BindCraft に渡す「素材」を作る役
  • BoltzGen 単体では「良いかどうかの評価」はしない
  • → 出力 PDB を BindCraft に流すのが典型

② BindCraft

■ 何ができるものか？

「このタンパク質は本当にターゲットに結合しそうか？」を評価・最適化するツール
  • 生成されたタンパク質とターゲットの
    • 結合スコア
    • 安定性
    • 形状の妥当性

      などを評価する

要するに：

2. 実際の内部動作（RFAntibody の例）

研究者が実行：

./run_rf_antibody.sh projectA/input.pdb

中身（例）：

#!/bin/bash

PROJECT_DIR=/data/projects/projectA

docker run --rm --gpus all \
  -v$PROJECT_DIR:/work \
  rf-antibody:1.0 \
  python run.py /work/input.pdb /work/output/

ここで何が起きているか？

  • -gpus all

    → GPU をコンテナに渡す

  • v /data/projects/projectA:/work

    → ホストとコンテナで 同じフォルダを共有

  • コンテナ内の /work に：
    • 入力ファイルが見える
    • 出力もそこに書かれる

結果はどこに出る？

/data/projects/projectA/output/

つまり：

コンテナは「計算だけして消える箱」

データは常にホスト側に残る

3. BindCraft も同じ思想

もし BindCraft が Docker 化されているなら：

./run_bindcraft.sh projectA/candidates/

内部：

docker run --rm --gpus all \
  -v /data/projects/projectA:/work \
  bindcraft:2.1 \
  python pipeline.py /work/candidates /work/results

結論

3つの中で「公式に Docker イメージ（または Docker 前提の配布形態）が明確に用意されているのは RFAntibody」です。

各アプリケーションの提供形態

① RFAntibody

  • ✅ Docker コンテナでの利用が前提・または公式に案内されている
  • ✅ GPU 利用も -gpus all 前提
  • ✅ 研究者は：
    docker run ...

    という形で実行する設計

つまり：

**RFAntibody は「コンテナとして使うアプリ」**という位置づけです。

実運用でも：

  • ホスト環境を汚さない
  • 依存関係を気にしなくてよい
  • 再現性が高い

という理由で、Docker で使うのがほぼ前提になります。

② BoltzGen

  • ❌ 公式に「この Docker イメージを使え」とはなっていない
  • ⭕ conda / pip 環境にインストールして使う前提
  • ⭕ 「新しい conda 環境を作れ」という案内が基本

つまり：

**BoltzGen は「Python 環境にインストールして使う研究コード」**という位置づけです。

※ 理論上は自作 Docker 化できますが、公式配布物ではありません。

③ BindCraft

  • ❌ こちらも 基本はソースコード + Python 環境（conda 等）前提
  • ❌ 公式に「これを使え」という Docker イメージが標準提供されているケースは一般的ではありません
  • ⭕ CUDA / PyTorch のバージョン指定が細かく、環境構築がシビア

つまり：

BindCraft も「ローカル（または VM）に環境を組んで使う研究コード」枠です。

特に AI / 機械学習 / 研究用途で非常によく使われています。

まず一言でいうと

「インストール不要で、ブラウザを開くだけで GPU 付き Python 環境が使えるサービス」

です。

何ができるのか？

  • ブラウザで：
    • Python を実行できる
    • Jupyter Notebook 形式でコードを書ける
  • 環境構築：
    • 不要（Google が用意してくれている）
  • しかも：
    • GPU や TPU も使える

研究者・学生・AIエンジニアが、

「とりあえず試す」

「論文のコードを動かしてみる」

用途で 世界中で使っている標準的な実験環境です。

見た目のイメージ

  • Webブラウザで：
    • 上：コードセル
    • 下：実行結果
  • いわゆる Jupyter Notebook とほぼ同じ操作感

なぜ研究者がよく使うのか？

理由1：環境構築が不要

普通は：

  • Python 入れて
  • CUDA 入れて
  • PyTorch 入れて
  • 依存関係で地獄を見る

ですが、Colab は：

最初からだいたい全部入っている

理由2：GPU がすぐ使える

  • 無料プランでも GPU が使えることがある
  • 有料（Colab Pro / Pro+）なら：
    • より高性能な GPU
    • より長時間実行

理由3：URL 1つで共有できる

  • Notebook を共有して：
    • 「このコードをここから実行してください」

      という運用ができる

ただし重大な制限（今回の案件の背景）

❌ 制限1：24時間で強制終了

  • セッションは：
    • 数時間〜最大24時間で切断
  • つまり：
    長時間計算が途中で強制終了される

❌ 制限2：環境は毎回リセット

  • 再接続すると：
    • インストールしたライブラリ
    • ダウンロードしたモデル
    • 全部消える（または再セットアップが必要）

❌ 制限3：Docker が基本使えない

  • RFAntibody のような：
    • 「Docker 前提ツール」が使いにくい／ほぼ使えない

❌ 制限4：性能・GPU が選べない

  • GPU は：
    • 何が割り当てられるか運次第
  • VRAM が足りなくて落ちることも多い

なぜ今回「Colab をやめたい」のか？

お客様の背景：

24時間でセッションが切れるコンテナが使えない（RFAntibody が動かない）

これは Colab の構造的制限そのものです。

つまり：

「実験環境」から「ちゃんとした計算サーバ」に移行したい」

という話です。

以下は Google Colab の有料プラン（Colab Pro / Colab Pro+）に関する最新のポイントと料金の目安です。

1) そもそも Colab の有料プランとは？

Google Colab はブラウザ上で Python 実行環境（Jupyter Notebook 形式）を使えるサービスで、有料版には Colab Pro / Colab Pro+ があります。kikagaku.co.jp+1

2) 24時間制限について（最重要）

24時間の実行制限はあるか？

はい。

  • 有料版（Colab Pro / Pro+）でも 1 セッションあたり最大約 24 時間までの連続実行制限があります。kikagaku.co.jp

注意点

  • 24時間は「理論上の最大値」であり、実際のセッションが 途中で切断されることがある（例えば 10～20 時間程度）という報告もあります。GitHub
  • 無料版は通常さらに短く、数時間〜12時間程度で切断されるという制約があるという整理もあります。kikagaku.co.jp

4) なぜ 24時間制限があるのか？

  • Colab は 共有 GPU リソースを割り当てるサービスであり、

    「ユーザーごとに無制限の専有リソース」を保証するものではありません。

  • したがって、長時間連続稼働の保証（例：48時間以上）はサービス仕様として提供されず、

    1セッションあたりの最大時間が設けられています。kikagaku.co.jp

5) Colab Pro / Pro+ の活用限界（今回の案件に照らして）

良い点

  • 手軽に GPU 付き Python 実行環境が使える
  • セッション時間や GPU が無料より長く・強くなる

制限

  • 最大でも 24 時間で強制終了される可能性は残る
  • Docker や長期バッチ処理には向かない（Colab は Notebook 実行前提）
  • 研究者が長時間走らせたい計算には制約がある

上記理由により、今回のように 「長時間バッチワーク・Docker コンテナ実行」 を想定する場合は、Colab 環境単体では要件を満たせないという背景があり、Azure などの専用サーバ／VM を提案する背景になっています。kikagaku.co.jp

① 時間スケールの問題（24時間制限）

❌ Colab では無理なケース

  • 1ジョブの実行時間が：
    • 24時間を超える
    • あるいは 10〜20時間以上を頻繁に超える
  • 例：
    • 大きなタンパク質設計ジョブ
    • 数千〜数万候補を評価するパイプライン
    • BindCraft の大規模スクリーニング
    • RFAntibody の複数ラウンド最適化

規模感の例

処理内容典型時間

中規模設計5〜20時間

→ 「1回の計算が1日を超える」時点で Colab は脱落です。

② 計算規模（GPUメモリ・性能）

❌ Colab では無理なケース

    • 24GB を超える
    • 40GB〜80GB クラスが必要
  • 例：
    • 大規模タンパク質複合体
    • 長鎖配列（>1000aa）
    • 高分解能設計

④ 環境再現性の問題

❌ Colab では無理なケース

  • 論文再現性が必須
  • バージョン固定が必須
  • 数か月〜数年同じ環境を維持したい

Colab は：

  • GPU が変わる
  • CUDA が変わる
  • 環境が勝手に変わる

→ **「去年と同じ実験を今年も全く同じ条件で」**がほぼ不可能。

⑤ Docker 前提ツール（RFAntibody 等）

❌ Colab では無理 or 非常に面倒

  • Docker 前提のツール
  • 複数コンテナを使うパイプライン
  • 自作コンテナで環境固定したい

→ Colab は「Docker を自由に使える計算サーバ」ではない

⑥ データ容量スケール

❌ Colab では無理なケース

  • モデル + データで：
    • 数百GB〜TB級
  • 例：
    • 大規模構造データベース
    • 大量の生成結果保存
    • 複数プロジェクト並行

Colab は：

  • ローカルディスクは一時領域
  • 再起動で消える

⑦ 並列実行・チーム利用

❌ Colab では無理なケース

  • 複数人で：
    • 同時に重いジョブを回したい
  • GPU を：

    • ジョブキュー管理したい

現在、お客様は Google Colab を用いて以下の研究用アプリケーションを利用している。

  • BoltzGen / BindCraft を GPU 環境で安定稼働

  • 研究者が「ジョブを投げて放置できる」運用
