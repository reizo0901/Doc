Microsoft 365（M365）：利用基盤全体（ID/端末管理/セキュリティ機能含む）
Defender for Endpoint（MDE）：端末防御（将来的に置換予定）
（参考）AD（オンプレ）：現行ID基盤がオンプレADの場合に登場
（参考）Entra ID（旧Azure AD）：クラウドID（M365側の認証基盤）
1. 全体の切替え方針（結論）
切替は大きく **3つの“本番切替点”**があります。
ファイル切替点（FS01 → FSx）：共有の参照先をFSxに変更する瞬間
メール切替点（EX01 → Exchange Online）：メール配送（MX/ルーティング）と利用者のメールボックスをEXOへ寄せる瞬間
セキュリティ切替点（SEP → MDE）：端末防御をMDEに寄せる瞬間（これは“段階的同居”が基本）
この3つを 同日にやらず、順番に進めます。
2. 段階的スケジュール（フェーズ別）と切替タイミング
Phase 0：準備（切替はしない／設計を固める）
目的：後工程の切替を“短時間”にするための準備
このフェーズで確定させること（切替に直結）

FS01の共有一覧（重要共有／部門共有／例外共有）と「Phase 1対象」を確定
FSx 側の共有設計（共有名、パス、容量、バックアップ、スナップショット）
アクセス権の付与方式（ご提示のとおり コマンドで一括適用）
メール移行方式（ハイブリッド／段階移行／移行バッチ運用）
M365導入範囲（上位層・経営層から開始、など）
Defender for Endpoint の展開方式（PoC→パイロット→展開）
切替点：なし（設計のみ）
Phase 1：先行移行（少人数・限定領域で“本番同等”を一回やる）
1) ファイル（FS01 → FSx）の先行切替
対象：上位層／経営層が使う“重要共有”だけ
手順と切替点

事前同期：FS01 → FSx を robocopy で実施（複数回）
差分が小さくなったら 静止点（短時間の書き込み停止）を取る
最終差分同期（robocopy）
ACLをコマンドでFSxに一括付与
切替点（Cutover-FS-P1）：
利用者がアクセスする共有パス（例：ドライブマップ／DFS／ショートカット）を FSx側へ向ける
切替後：FS01はRead Only化（過去バージョン参照用途）
ここで重要なのは「FS01を残す」が目的ではなく、
“復旧の保険（過去バージョン参照）”としてRead Onlyで保持する点です。
2) メール（EX01 → Exchange Online）の先行切替
対象：上位層／経営層のみ
手順と切替点（代表パターン）

M365側にユーザー・ライセンス準備
移行バッチ（メールボックス移行）を実行し、対象者のメールボックスをEXOへ
切替点（Cutover-Mail-P1）：
対象者の“実利用”がEXOになる（Outlook再構成／Autodiscover切替等）
メール配送（社外→社内）のルーティングは、Phase 1では「全社切替」せず、段階運用も可能
（ただし設計により、MX切替をPhase 2にまとめる方が分かりやすいケースが多い）
3) セキュリティ（Defender for Endpoint）の先行導入
対象：先行移行ユーザーの端末のみ
MDEをパイロット導入（監視中心→ブロック有効化は段階的）
切替点（Cutover-MDE-P1）：
「監視開始日」または「ブロック開始日」を明確に分けて宣言する
（例：監視開始=当日、ブロック開始=2週間後）
Gate：評価（ここが“コスト調整ポイント”）
ここで 次の拡大をするかを判断します。
FSx：性能・運用・バックアップ/復旧が回るか
EXO：移行後の問い合わせ量、誤送信・PPAP対策の運用負荷
MDE：誤検知、端末負荷、運用体制
判断結果
問題なし → Phase 2へ（対象を増やす）
課題あり → 設計修正（この時点で拡大しない＝コスト膨張を止める）
効果薄 → 先行範囲で止める（投資を限定できる）
Phase 2：拡大（全社へ波状展開。切替点を“複数回”作る）
1) ファイル（FS01 → FSx）の波状切替
対象：部門共有を順に
部門A：事前同期→静止点→最終同期→ACL→Cutover-FS-P2-A→FS01の該当共有をRead Only
部門B：同様に Cutover-FS-P2-B
…を繰り返す
ポイント
切替点が「部門単位」で複数回発生する（＝段階的）
FS01のRead Only化も「共有単位」で進む
2) メール（EX01 → Exchange Online）の全社切替（ここで“配送”を統一）
対象：一般部門
ユーザー移行（バッチ）を段階実施
切替点（Cutover-MX/Route-P2）：
原則ここで 社外メール配送の主経路（MX/ルーティング）をEXO側へ統一
※実装方式（Hybrid/Connector/Transport）でやり方は異なりますが、提案上は「このタイミングで配送の中心がEXOになる」と宣言すると分かりやすいです。
旧EX01は“残存ユーザー/アーカイブ”がいなくなった段階で縮退
3) Defender for Endpoint（MDE）の部門展開
端末を部門単位で展開
切替点（Cutover-MDE-P2-Dept）：
「監視→ブロック」の順で切替日を2段階で宣言
SEP等の旧AVは、共存可否を確認しつつ段階的に縮退（ポリシーで制御）
Phase 3：廃止・最適化（“残す”から“消す”へ）
ここが「コストを下げ切る」フェーズです。
1) FS01（旧ファイルサーバ）の廃止
全共有がFSxへ切替済み
Read Onlyとしての保持期間を満了（例：○ヶ月）
切替点（Decom-FS01）：FS01停止・廃止（バックアップも停止）
2) EX01（旧メール）の廃止
残存メールボックス/配送がゼロ
切替点（Decom-EX01）：EX01停止・廃止
3) M365のライセンス最適化
実利用に基づき、過剰ライセンスを是正
“コスト調整の最終弁”：必要な層だけ上位ライセンスを維持
3. 「切替タイミング一覧」だけを抜き出す（最重要）
Cutover-FS-P1：重要共有（上位層/経営層）を FSxへ向ける日
同日：FS01の該当共有を Read Only化
Cutover-Mail-P1：上位層/経営層のメールボックスが EXO利用開始する日
Cutover-MDE-P1：先行端末で MDE監視開始日／ブロック開始日（2段階）
Cutover-FS-P2-各部門：部門共有を順次 FSxへ向ける日（部門単位で複数回）
Cutover-MX/Route-P2：社外配送（MX/主ルーティング）を EXO中心へ統一する日
Cutover-MDE-P2-各部門：MDEを部門単位に展開し 監視→ブロックへ切替
Decom-FS01：FS01を停止・廃止する日（保持期間満了後）
Decom-EX01：EX01を停止・廃止する日（残存ゼロ確認後）
4. 段階的にすることで「コストを調整できる」ことの言い切り方
Phase 1では FSxの容量・M365ライセンス・MDE展開台数を最小にするため、初期費用/運用負荷を抑えられる
Gateで評価し、問題があれば 拡大を止めて設計を直せる（＝手戻りコストの爆発を回避）
Phase 2に進む場合も、部門単位で切替するため、必要な分だけコストを段階増できる
Phase 3で旧環境を廃止することで、二重コストを確実に終わらせられる
